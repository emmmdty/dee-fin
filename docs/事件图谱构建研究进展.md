## 研究结论总览

### 执行摘要

围绕“事件图谱构建”而不是普通事件抽取来严格筛选后，我没有发现 **15 篇以上都以“完整事件图/事件知识图谱构建”为直接目标** 的 2024-01-01 至 2026-07-20 论文；更接近真实研究版图的划分是：**直接核心构建论文约 10 篇**，另有 **8–9 篇核心构建模块/紧邻论文** 主要集中在跨文档事件共指、事件链接、多事件关系联合建模、事件本体/Schema 归纳与图补全。这说明 2024 年以来该领域确实升温，但“端到端、可验证、可追溯、跨文档、全图一致”的成熟研究仍然不多，不能用普通 trigger/argument 抽取论文凑数。citeturn36view0turn38view0turn26search0turn25view0turn35search4turn33search2turn33search15

2024 年以来最清晰的主线有六条：**长文档事件图生成**、**跨文档事件对齐/共指/聚合**、**多事件关系联合建模与全局一致性**、**事件中心知识图谱构建与融合**、**Schema/ontology 归纳**、以及 **event graph completion / verification / repair / incremental update**。其中，前四条已经形成相对稳定的研究方向；Schema/ontology 归纳仍在持续，但 2024 年后产出不算高；“验证—修复—增量更新”则更像新兴方向，论文数量明显少于生成、聚合和关系建模。citeturn38view0turn26search0turn29search5turn8search1turn9view0turn35search4turn32search3turn31search0turn33search15

方法上，2024 年后的真正变化，不是简单把 BERT 换成 LLM，而是从 **pairwise 判别** 转向 **global graph modeling** 与 **graph generation**，从 **单文档局部预测** 转向 **跨文档聚合与事件身份统一**，从 **局部分类 F1** 转向 **更关注全图一致性、图补全、约束优化和结构修复**。LLM 的真实收益主要体现在：长上下文聚合、零样本/弱监督图生成、跨文档多步推理、自动生成远程监督或 rationale；但它同时带来了 hallucination、边缺失、关系不满足逻辑约束、证据链不显式等问题，因此 2025–2026 的代表性工作已经开始加入 **iterative refinement、constraint optimization、logic-induced graph、LLM+SLM 协作**。citeturn36view0turn38view0turn21search5turn29search5turn25view0turn8search1turn9view0turn33search15

如果要把这一主题设计成硕士论文的两个研究章节，最有价值、也最适合 **公开数据 + 开源模型 + 1–2 张 RTX 4090** 的方案，不是把同一模型拆成两个工程模块，而是做成一个递进链条：**第一章做跨文档事件聚合与证据对齐，第二章做基于约束的多关系事件图构建与自动修复**。第一章解决“同一事件是谁、证据来自哪里、如何得到统一事件节点”；第二章在此基础上解决“这些节点之间如何形成时间/因果/子事件等结构且保持全图一致”。这套设计既能独立投稿，也有天然的前后章支撑关系。citeturn26search0turn29search5turn25view0turn8search1turn9view0turn21search5turn31search0turn33search2

### 事件图谱构建的概念与研究边界

本报告采用的“事件图谱构建”定义是：**从文本、已有事件记录或多源数据中，识别并组织事件节点、参与实体、事件属性、事件间关系及其证据，形成带有明确节点类型、边类型与结构约束的事件图或事件知识图谱**。这一定义与普通事件抽取不同，后者通常只到 trigger/argument；而事件图谱构建至少还要解决 **事件身份统一、关系组织、跨文档聚合、图结构输出、全局约束或证据回溯** 中的一部分。Set-Aligning、CALLMSAE、CDEE、MEET、ChronoGrapher、EventRAG 与 PLAF 等论文都把输出目标定位到了图、跨文档聚合图、事件中心 KG 或图补全，而不是单点抽取。citeturn36view0turn38view0turn26search3turn25view0turn35search4turn33search0turn33search2

### 文献检索方法与纳入排除标准

检索范围覆盖用户指定时间窗 **2024-01-01 至 2026-07-20**，主要来源包括 ACL Anthology、arXiv、Semantic Web Journal / SAGE 页面、IJCAI、EMNLP、NAACL、ACL、COLING 以及作者官方 GitHub。检索关键词与用户给出的组合一致，包括 “event graph construction”, “cross-document event extraction”, “event temporal graph generation”, “event graph completion”, “event-centric knowledge graph”, “event schema induction”, “event linking”, “LLM event graph”等，并对论文页、仓库页和数据页进行链接核验。citeturn36view0turn38view0turn26search0turn21search5turn35search4turn31search0turn33search15

纳入标准有三条：其一，**输出必须达到图级或准图级结构**，包括事件节点、关系边、跨文档聚合节点、补全节点或面向图的一致性优化；其二，**论文主题必须正面服务于事件图谱构建**，如跨文档事件共指、事件链接、多关系联合建模、Schema 归纳、图补全；其三，**优先正式发表论文，其次公开 arXiv 预印本**。排除标准也很严格：只做触发词抽取、只做事件分类、只做下游问答/预测而不研究构建、或只是标题带 event 但不处理事件语义结构的论文，均不作为核心论文。citeturn25view0turn26search3turn13search12turn31search0

最终筛选结果是：**直接核心构建论文 10 篇**，**核心模块/紧邻论文 9 篇**，另补充少量 2024 年前奠基论文作为任务定义、经典基线和数据来源。这个结果本身就是一个研究判断：2024 年以来，事件图谱构建的“研究面”明显扩大，但真正把“事件节点 + 关系边 + 跨文档身份 + 结构约束/验证 + 图输出”串成完整体系的论文数量，依然有限。citeturn36view0turn38view0turn26search0turn25view0turn33search0turn35search4turn33search2turn33search15

**长文档事件图生成** 之所以已经是主流，是因为 2024–2025 连续出现了三类清晰方法：Set-Aligning 把序列生成改造成 set-aware graph generation；CALLMSAE 则把 salient event 选择与边修复引入事件图生成；Global Zero-shot 更进一步，直接一次性生成完整 temporal graph，再做 constraint optimization。这条线的优点是问题定义清楚、图输出直接、实验闭环完整；缺点是当前最成熟的仍是 temporal graph，relation space 还不够全。citeturn36view0turn38view0turn21search5turn21search8

**跨文档事件对齐与融合** 已经形成稳定方向，而且是当前最适合做硕士论文的方向之一。CDEE 把 cross-document event extraction 正式化；SECURE 把 LLM 引入跨文档共指；MEET 进一步把问题提升为多文档事件聚合，新增 ECB++ 与 CEAF-MEE，明确指出单文档只覆盖同一真实事件约一半左右的论元信息。这条方向的研究价值很高，因为它直面事件图谱构建最关键的对象级问题：**一个“事件节点”到底该如何跨来源定义并携带证据**。citeturn26search3turn29search5turn25view0

**多事件关系联合建模** 也已经是主流。MAVEN-ERE 的发布使 temporal/causal/subevent/coreference 的联合建模真正可做，GraphERE 强调 graph-enhanced event embeddings 和 task-specific dynamic graph，LogicERE 强调逻辑诱导高阶推理，MMD-ERE 与 GDLLM 则代表 LLM/agent 化的新范式。这个方向最强的价值是能显式回答“图的边如何构建”，但最大痛点是多数工作仍停留在 **给定事件节点、预测事件对关系**，还未和跨文档聚合完成统一。citeturn39search1turn8search1turn9view0turn27search2turn30search2

**事件中心知识图谱构建与图融合** 目前已经有代表作，但论文总量仍少于前两条。EventRAG 把多文档事件抽取、等价节点合并和 under-connected relation expansion 放进同一框架；ChronoGrapher 则表明，除了“从文本抽”之外，还可以“从已有通用 KG 重构事件中心图”。如果用户目标是做一个更接近“图谱系统”的硕士论文，这条线很有吸引力；但对单卡/双卡环境而言，其实验可控性略低于纯文本跨文档聚合。citeturn33search0turn35search4turn35search18

## 方法、数据、代码与LLM影响

### 方法论演进

从 2024 年以后，事件图谱构建在方法论上最明显的变化，是从 **pipeline 的局部最优** 逐步转向 **global graph modeling / generation 的全图视角**。CDEE 仍是典型 pipeline：抽取、共指、规范化、角色对齐逐步推进；MEET 和 SECURE 则把 LLM 放到跨步骤推理层，让小模型负责局部可学习子任务，LLM 负责全局聚合判断；Set-Aligning、CALLMSAE、Global Zero-shot 则直接把目标表述成 graph generation，而不再只是 pairwise classification。citeturn26search3turn25view0turn29search5turn36view0turn38view0turn21search5

**Pairwise classification 与 global graph modeling** 的差别，已经不再是表述技巧，而是性能瓶颈所在。GraphERE 仍然保留 relation-level 预测，但通过 task-specific dynamic graph 引入图结构；LogicERE 更进一步，把逻辑约束嵌入 logic constraint induced graph，再用高阶推理捕获 event/event-pair 的耦合；Global Zero-shot 则干脆生成整个 temporal graph 后再做 constraint optimization。这三种工作共同表明：只在事件对上做独立分类，很难保证最终图的对称性、传递性和全局一致性。citeturn8search1turn9view0turn21search5

**Transformer 与 GNN/graph scaffold 的结合** 仍然有效，但“只换 backbone”已经不够了。GraphERE 的贡献点并不只是编码器，而是 graph-enhanced event embeddings 中把 arguments、AMR/IE 图和多任务动态边结合起来；GDLLM 也不是简单把 LLM 用到 temporal relation extraction，而是构造 distance-aware graph 并配 GAT 辅助长距离依赖。换言之，2024–2026 里真正有价值的 paper，不是在“谁用了更大的模型”，而是在“谁把结构先验真正塞进了表示与推理过程”。citeturn8search1turn30search2

**判别式方法与生成式方法** 的边界也在变化。判别式方法在 relation extraction、coreference resolution 上仍有强势基线，特别是当训练数据清晰、标签定义稳定时；但生成式方法已经在两个地方显示出明显优势：第一，长文档图生成，如 Set-Aligning 与 CALLMSAE；第二，跨文档多步推理，如 SECURE 与 MEET。问题在于，生成式方法同时引入了 hallucination、边遗漏、输出格式不稳定、证据不完整等新风险，因此 CALLMSAE 采用 iterative code refinement，Global Zero-shot 采用 temporal constraint optimization，MEET 则让 LLM 不直接替代所有模块，而是与 SLM 协作。citeturn36view0turn38view0turn29search5turn25view0turn21search5

就 **小模型与 LLM** 而言，2024 之后的经验非常一致：在事件图谱构建里，LLM 的长项是 **全局叙事理解、弱监督数据构造、rationale、salience 识别、multi-step reasoning**；小模型的长项是 **稳定训练、低成本微调、局部分类精度与可控解码**。SECURE、MEET、Okay 都采用了某种形式的 LLM-to-SLM 协作或蒸馏，这比“全部交给闭源 API”更适合学位论文场景。citeturn29search5turn25view0turn28search2

关于 **Schema-constrained generation / constrained decoding / generate-verify-repair loop**，目前还没有完全统一的范式，但趋势很明确。Set-Aligning 用 set-property regularization 修正线性化图的损失偏差；CALLMSAE 用 refinement 降 hallucination；Global Zero-shot 用 temporal constraints 做后验优化；LogicERE 在训练和图结构里显式编码逻辑；SchemaEGC 与 PLAF 则把 schema / prompt 作为图补全先验。这些工作共同说明：未来事件图谱构建的关键，不是“再生成一次”，而是“生成后如何验证、如何修补、如何保证图有效”。citeturn36view0turn38view0turn21search5turn9view0turn31search0turn33search2

#### 评价指标

评价指标也清楚暴露了当前研究的局限。跨文档事件对齐工作通常用 **MUC / B³ / CEAF_e / CoNLL F1**；MEET 为多文档聚合新增了 **CEAF-MEE**，说明原有文档级事件抽取指标无法衡量“多个聚合事件”的结构匹配。关系建模工作则主要报告 relation-level F1；LogicERE 和 GraphERE 的对比依然是 TRE/SRE/F1；Global Zero-shot 开始把 **全图一致性与 constraint optimization** 纳入评测逻辑。换句话说，**统一的端到端事件图评价指标仍然缺失**，尤其缺少兼顾事件节点、事件边、证据与全局一致性的单一指标。citeturn25view0turn9view0turn8search1turn21search5

### 开源代码与复现条件

从复现角度看，2024–2026 的事件图谱构建研究出现了明显分层。**复现条件最好** 的一层，是公开了代码、训练脚本甚至数据处理脚本的工作，包括 Set-Aligning、CALLMSAE、SECURE、MEET、DIE-EC、llama_cdcr、multimodal-coreference、CLES、GlobalZeroShotTRE、CEOEventOntology、ChronoGrapher 和 PLAF。这些工作大多依赖公开 PLM/LLM，或者只在推理中使用开源模型，因此比较适合硕士阶段复现与再研究。citeturn37view0turn19search9turn29search1turn25view1turn29search10turn28search11turn29search12turn26search2turn21search2turn32search7turn35search18turn33search2

**复现条件中等** 的一层，是论文页清楚、任务清楚，但官方代码并不稳定公开或在本次检索中未见明确链接，例如 EventRAG、GraphERE、LogicERE、MMD-ERE、TingIS。它们仍可做“方法复现”或“思路迁移”，但更适合作为 baseline 思想而不是完全依赖原代码。citeturn33search0turn8search1turn9view0turn27search2turn33search15

如果把算力限定在 **1–2 张 RTX 4090**，最现实的选择不是直接微调超大闭源 LLM，而是：在章节一使用 **Longformer / RoBERTa / DeBERTa / Qwen2.5-7B LoRA** 这类模型做跨文档聚合；在章节二使用 **T5-base / Flan-T5 / RoBERTa + GNN / 7B 级开源 LLM + constrained decoding** 做图生成或修复。这是对公开论文所用基模和官方仓库训练流程的保守推断，也最符合学位论文的可控性。citeturn37view0turn25view0turn29search1turn21search2turn33search2

### LLM对事件图谱构建的影响

LLM 对该领域的最大正面影响，不是“单点关系分类更高了”，而是 **长文档、多文档、弱监督、结构生成** 这些过去难以统一处理的环节被打通了。CALLMSAE 证明 LLM 可以先做 summary-based salient event identification，再对图边做 iterative refinement；SECURE 和 MEET 证明 LLM 可以在跨文档事件聚合中承担多步推理和全局决策；Global Zero-shot 则表明，LLM 完全可以从 pairwise 分类器提升为全图生成器。citeturn38view0turn29search5turn25view0turn21search5

但 LLM 带来的问题同样真实。CALLMSAE 之所以要显式做 code refinement，是因为 graph generation 会产生 hallucinated relations 和 missing edges；MEET 明确指出 sequence-to-sequence LLM 往往缺乏显式 rationale，且面对多个文档、数百事件时，很难在单一序列里稳定完成复杂聚合；Global Zero-shot 之所以要做 temporal constraint optimization，也是因为 LLM 原始生成并不能自然保证 temporal consistency。citeturn38view0turn25view0turn21search5

因此，2024–2026 的一个关键信号是：**LLM 已经进入事件图谱构建主流程，但几乎没有代表作把它当成“裸生成器”直接放行**。真正有效的方法都加了约束或协作层：Set-Aligning 用 set regularization，SECURE/MEET 用 LLM+SLM 协作，LogicERE 用 logic constraints，GDLLM 用 distance-aware graph，PLAF 用 graph prompt learning。换言之，LLM 的收益是真实的，但必须被 **schema / logic / graph scaffold / retrieval / verification** 驯化。citeturn36view0turn29search5turn25view0turn9view0turn30search2turn33search2

## 最终推荐方案、风险与链接汇总

### 最终推荐的两章方案

我明确推荐 **方案甲：跨文档事件聚合与证据对齐 → 约束驱动的多关系事件图构建与自动修复**。原因有四个。第一，它最贴近 2024–2026 文献真正暴露出来的研究断点：节点聚合和关系一致性仍分别被不同论文处理，尚未被打通。第二，它的两章边界清楚：第一章解决“图的点”，第二章解决“图的边与图合法性”。第三，它几乎完全建立在公开数据和开源代码上，不依赖闭源 API。第四，它对于 1–2 张 RTX 4090 非常现实。citeturn25view0turn26search3turn29search5turn9view0turn21search5turn33search2

其他方案不是首选，不是因为不好，而是因为它们各有短板。方案乙的问题在于第二章“多图融合与 event-centric KG 评测”公开标准不足，容易把硕士论文拖进系统工程；方案丙则更稳，但它的第二章过于像“补全后处理”，创新叙事略弱。相比之下，方案甲既保留了方案丙的可控性，又保留了方案乙的图级研究深度。citeturn35search4turn33search0turn31search0turn33search2

在这套推荐方案里，**第一章最核心的贡献** 应该是：提出一种 **evidence-aware canonical event representation**，把跨文档事件聚合从“事件共指”提升为“统一事件节点构建”，输出不只是 cluster，而是带有 canonical trigger、聚合论元、来源证据和置信度的事件节点。这个贡献点直接命中文献中“事件是谁、证据在哪里”没有统一解决的问题。citeturn25view0turn26search3turn29search5

**第二章最核心的贡献** 应该是：提出一个 **multi-relation constraint-and-repair framework**，在第一章统一好的事件节点之上，同时预测 temporal/causal/subevent/coreference 边，并显式报告与优化图级 consistency violation。与其重复做“更好的 relation classifier”，不如把重点放在 **graph legality、repair provenance、constraint-aware re-ranking** 上，这样更符合 2025–2026 的前沿趋势。citeturn9view0turn21search5turn31search0turn33search2

两章之间的逻辑关系非常自然：**没有可靠的 canonical event nodes，就不可能稳定地构造跨文档多关系事件图；而没有第二章的多关系构图与修复，第一章的跨文档聚合只能停留在“事件表”，还不是“事件图谱”**。所以这不是两个松散工作，而是从节点级构建走向边级与全图级构建的递进。citeturn25view0turn26search3turn9view0

最推荐的数据集组合是：第一章用 **ECB++ + CLES** 做主实验，**ECB+ / GVC / MCECR** 做补充共指实验；第二章用 **MAVEN-ERE + OmniTemp** 做多关系与一致性实验，再把第一章输出的 canonical events 派生为自己的图级验证集。这样既能利用现有公开数据，又能保证两章之间的数据接口真实存在。citeturn25view0turn26search3turn39search1turn21search16turn40search20turn40search16

最推荐的基线组合是：第一章对比 **SECURE、MEET、DIE-EC、Okay、ACCI**；第二章对比 **GraphERE、LogicERE、MMD-ERE、GDLLM、GlobalZeroShotTRE、PLAF**。这些基线覆盖了 pipeline、LLM+SLM 协作、rationale 蒸馏、graph-enhanced relation modeling、logic constraints、global generation 和 graph completion 等不同技术流派。citeturn29search5turn25view0turn29search10turn28search2turn27search3turn8search1turn9view0turn27search2turn30search2turn21search5turn33search2

最关键的评价指标应明确分层。第一章以 **CoNLL F1 / B³ / CEAF_e + argument aggregation F1 + evidence attribution precision/recall** 为主；第二章以 **relation F1 + graph consistency violation rate + repair gain** 为主；如果精力允许，再增加一个简单的 **graph edit 或 graph overlap** 指标作为整图补充。这里最重要的不是追求“单一总分”，而是显式证明第二章确实改善了图的整体合法性。citeturn25view0turn9view0turn21search5

最可能失败的环节主要有两个。第一，第一章的“证据对齐”如果没设计好，可能最后只得到一个普通共指模型；第二，第二章的 repair 如果只带来很小收益，容易被质疑为复杂化系统。这两个风险都可以通过替代设计缓解：第一章若证据标注压力过大，可先做 **canonical event table**；第二章若 repair 收益不稳，可退而做 **consistency-aware reranking / constrained decoding**。citeturn25view0turn21search5turn33search2

### 风险与替代方案

从学位论文管理角度，方案甲的最大风险不是算力，而是 **评测协议设计**。第一章需要证明“证据对齐”不是装饰；第二章需要证明“图修复”不是规则堆砌。因此，最稳妥的策略不是增加大量人工标注，而是优先利用已有数据定义可自动计算的指标，再在很小规模样本上做人工核验。MEET 的 CEAF-MEE 与 Global Zero-shot 的 constraint optimization，已经提供了很好的出发点。citeturn25view0turn21search5

如果最终发现第一章比预期更难，可以退一步，把第一章收缩为“**跨文档 event canonicalization with evidence**”；如果第二章比预期更难，则可改成“**multi-relation graph reranking with logic constraints**”，不必执着于从零做完整 repair 系统。这样仍能保持两章独立且递进。citeturn25view0turn9view0turn33search2
